How DutchMate Works
01

From Hover To Tooltip

Start with the user experience: you hover a Dutch word, and a tiny translation appears without breaking your reading flow.

What this app actually is

DutchMate is a browser extension for people reading Dutch websites while thinking in English and Telugu.

The key promise is simple: keep reading the web, and let translation come to you instead of forcing you into a new tab.

🖱️

Hover mode

Pause over a single word and DutchMate quietly checks whether a tooltip should appear.

✂️

Selection mode

Select a short phrase or sentence when one word is not enough.

📚

Learning first

This is not a generic translator. It is tuned for a Dutch, English, Telugu learning triangle.

💡
Why this matters for steering AI

If you can describe the product as "a context-aware translation tooltip inside a browser extension," AI tools stop suggesting the wrong architecture, like building a full website or chat app instead.

The core journey is a relay race

Think of DutchMate like a relay team. Each runner touches the baton briefly, does one job, and passes it on fast.

1
You hover or select text

The web page is the track. DutchMate notices a real reading action, not a random background event.

2
The content script packages the request

It captures the text, location, and context like hover or selection.

3
The background worker asks for translation

That keeps network work out of the page-facing layer.

4
The tooltip renders the answer

The reader sees a tiny result right where their eyes already are.

Translation And Cache

This is the engine room: provider choice, fallback behavior, and the little memory system that saves time and money.

The engine room has two jobs

Imagine a ferry terminal. One worker decides which boat to use. Another remembers that you already bought a ticket. DutchMate splits those same concerns.

🧭

Provider choice

Use the configured endpoint if one exists. Otherwise fall back to a placeholder path that keeps the extension testable.

⚡

Fast memory

A short-lived cache avoids doing the same translation twice in the current session.

🧳

Persistent memory

A longer-lived cache keeps useful results between sessions for selected single-word lookups.

The provider decision is tiny on purpose

This is a great example of a good seam. [src/translation/translation-service.ts](/home/mgurram/MGurramAI/projects/dutchmate-proj/dutchmate/src/translation/translation-service.ts:14) does not know how every provider works. It just decides which door to open.

export class TranslationService {
  private readonly placeholderProvider = new PlaceholderTranslationProvider();

  constructor(
    private readonly readSettings: () => Promise<TranslationProviderSettings>,
    private readonly cache: TranslationCache,
    private readonly persistentCache?: PersistentTranslationCacheLayer,
  ) {}

  async translate(request: TranslationRequest): Promise<TranslationResult> {
    const settings = await this.readSettings();

    if (!settings.providerEndpoint) {
      return this.placeholderProvider.translate(request);
    }

    return new CustomEndpointTranslationProvider(
      settings,
      this.cache,
      this.persistentCache,
    ).translate(request);
  }
}

async get(request: TranslationRequest): Promise<TranslationResult | null> {
  if (!this.shouldPersist(request)) {
    return null;
  }

  const data = await this.readData();
  const freshData = this.removeExpiredEntries(data);
  const entry = freshData.entries[getPersistentCacheEntryKey(request)] ?? null;

  if (freshData !== data) {
    await this.writeData(freshData);
  }

  return entry?.result ?? null;
}

Backend Guardrails

The backend is intentionally small, but it still protects itself with clear routes, health checks, and overload controls.

A small service still needs a bouncer

Think of the backend like a small venue. It needs a front door, a sign for staff to confirm it is open, and a bouncer who stops the room from becoming chaos.

1
GET /health says the venue is alive

Monitoring and debugging can ask whether the service is up without doing a real translation.

2
POST /translate is the main work entrance

Translation requests go through one predictable contract.

3
Rate limits and backpressure protect the room

If too many requests arrive, the server says so clearly instead of silently melting down.

🛡️
Universal backend lesson

Most reliability work is not "do the happy path." It is "fail early, fail clearly, and recover without poisoning the rest of the system."

return createServer(async (request, response) => {
  setCorsHeaders(response);

    sendEmpty(response, 204);
    return;
  }

  if (request.method === "GET" && request.url === "/health") {
    sendJson(response, 200, {
      ok: true,
      service: "dutchmate-backend",
      runtime: translateRequests.runtimeSummary.snapshot(),
    });
    return;
  }

  if (request.method !== "POST" || request.url !== "/translate") {
    sendJson(response, 404, {
      error: "Use POST /translate",
    });
